# OpenTenBase 修复路径决策说明

日期：2026-05-17

主题：为什么这次不能只靠 RPM 安装后修配置，为什么最终必须采用源码修复并重编关键二进制

## 结论

本次 OpenTenBase 单机集群启动问题，最终不能通过“仅使用官方 RPM 安装包 + 调整配置”解决，必须采用“源码修复 + 重新编译关键二进制”的方案。

原因不是偏好源码，而是当前问题已经落在 OpenTenBase 运行时二进制行为本身，不是普通部署参数可覆盖的问题。

最终数据库正常启动，是在以下条件同时满足后实现的：

- 采用源码编译产物替代关键运行二进制
- 修复 GTM 启动阶段的线程绑核致命退出逻辑
- 保留低内存运行参数，适配当前小规格服务器

## 背景

目标服务器：

- `47.108.249.115`
- 约 `2 vCPU`
- 约 `1.8Gi` 内存
- `4Gi` swap

目标形态：

- OpenTenBase 5.0 单机集群
- `GTM + 1 CN + 1 DN`

本次优先级最高的目标不是先联调业务，而是先确保数据库本身可以稳定启动。

## 为什么仅靠 RPM 不够

### 1. RPM 路线存在运行时包装问题

在 RPM 路线下，已经实际观察到以下问题：

- `opentenbase_ctl` 找不到 `libssh2.so.1`
- 打包 `psql` 存在动态库混链问题
- 部分工具依赖包内库和系统库的组合关系不稳定

这类问题不属于 SISM 项目配置错误，而属于 OpenTenBase 当前 RPM 发行物的运行时包装问题。

### 2. RPM 路线的核心阻塞不是配置，而是 GTM 二进制行为

在 RPM 版本的 GTM 启动日志中，核心致命错误固定收敛为：

- `binding threads failed for 22`

其表现是：

- GTM 进入启动流程
- 刚开始监听端口
- 随后在线程绑核阶段直接 `FATAL` 退出

这是 GTM 程序内部把 `pthread_setaffinity_np()` 的失败视为致命错误导致的。

这种问题不是改 `DB_URL`、改端口、改普通配置项可以解决的，因为故障点已经进入可执行程序内部逻辑。

### 3. RPM 路线已经做过足够的配置级尝试

在决定切源码之前，已经实际完成过以下排查与修复：

- 修复单机 `localhost` 免密 SSH
- 补齐 `pgxc_ctl` 依赖的命令可见性
- 改用 `pgxc_ctl` 的单机 `GTM + CN + DN` 标准路径，而不是手工裸起 coordinator
- 将 `CN/DN` 的连接数、并发数、工作内存、结果缓存、workfile 等参数压到低内存可用范围
- 修复 `max_wal_senders must be less than max_connections`

这些动作都证明了一点：

- 不是不会部署
- 不是不会配参数
- 而是 GTM 二进制本身在当前环境下会因为线程绑核逻辑主动退出

## 为什么源码修复是必要的

### 1. 最终成功启动依赖了对 GTM 源码的修复

源码修复的关键点只有一个，但它决定了成败：

- 将 GTM 中线程绑核失败的处理方式
  从 `elog(ERROR, ...)`
  改为 `elog(WARNING, ...)` 并继续运行

也就是说：

- 旧逻辑：绑核失败就终止 GTM
- 新逻辑：绑核失败仅记告警，不阻断 GTM 正常启动

在当前云主机环境中，这个行为差异直接决定数据库是否能起来。

### 2. 源码编译后，GTM 的行为发生了实质变化

修复后的源码版 GTM 已验证表现为：

- 不再因 `binding threads failed for 22` 直接退出
- 能进入 `GTM_IN_PRODUCTION`
- 能持续存活
- 能支撑 `CN` 和 `DN` 后续启动

这不是配置偶然成功，而是二进制行为被明确改变后的结果。

### 3. 数据库最终正常启动，是源码修复结果

最终验证通过的运行状态：

- `GTM` 监听 `6666`
- `CN` 监听 `30004`
- `DN` 监听 `20008`
- 通过 `CN` 已可执行 SQL
- `pgxc_node` 中已可看到 `gtm / cn001 / dn001`

因此，“数据库已正常启动”这一结果，应归因于：

- 源码修复后的新二进制

而不是原始 RPM 包直接修复成功。

## 对团队/管理的建议

### 建议 1：对本次问题的正式表述

建议统一对内表述为：

> 本次 OpenTenBase 单机集群无法启动的根因，不是部署脚本或普通配置项错误，而是当前 RPM 发行物中的 GTM 运行时行为在目标环境下存在致命退出问题。经过配置级修复仍无法闭环后，最终采用源码修复并重编 GTM 关键二进制，成功完成数据库启动。

### 建议 2：后续环境策略

建议后续区分两种路线：

- 面向快速试用：继续评估官方 RPM 包
- 面向稳定落地：优先保留本次源码修复后的可运行产物和操作手册

### 建议 3：对外部维护方的反馈重点

建议重点反馈以下方面，而不是只说“起不来”：

- GTM 在线程绑核失败时不应直接 `FATAL`
- 单机 `pgxc_ctl` 模式对 `localhost SSH` 的依赖需要明确写入文档
- RPM 发行物在工具链、动态库加载、单机集群引导上仍有明显改进空间

## 最终判断

这次问题的答案是明确的：

- 如果坚持“只允许官方 RPM 原包，不修改运行二进制”，则无法稳定得到当前已验证成功的数据库启动结果
- 如果目标是“必须把数据库真正启动起来”，则本次必须采用源码修复方案

因此，本次“源码修复并重编关键二进制”不是技术偏好，而是达到数据库正常启动目标的必要手段。
